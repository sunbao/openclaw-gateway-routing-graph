# OpenSpec

## Title

Policy Control and Risk Governance for OpenClaw Gateway Routing Graph

## Status

- 状态：Draft
- 开发触发条件：**GitHub Star > 100**
- 当前阶段：需求冻结前的公开草案

## Background

当前项目已经实现了“链路可见性”：

- 可以看到入口消息
- 可以看到出口消息
- 可以看到会话、智能体、工具和渠道之间的流转
- 可以对链路做图形化展示和实时观察

但现阶段项目还主要是“观察型系统”，并不负责真正的策略执行。

下一步需求已经逐渐明确：不仅要“看见风险”，还要“理解风险”，最终具备“控制风险”的能力。

## Problem Statement

使用方希望通过该系统逐步具备以下能力：

- 入口关键词拦截
- 黑名单拦截
- 隐私泄露告警
- 出口高风险内容阻断
- 审核后放行

同时，又必须保持独立项目形态，避免与 OpenClaw 主工程形成紧耦合。

## Goals

本规格的目标是定义一个独立演进路径，使项目从“可视化监控”演进为“监控 + 告警 + 策略控制”系统。

具体目标：

- 在入口和出口两个方向识别风险
- 提供统一的规则模型
- 提供统一的命中动作模型
- 支持告警、拦截、审核和放行
- 保持独立部署和独立升级能力

## Non-Goals

本阶段不追求：

- 直接修改 OpenClaw 主工程源码
- 一步到位实现完整风控平台
- 对所有历史规则格式做兼容
- 默认存储和展示完整敏感正文

## Primary Use Cases

### Use Case 1：出口隐私泄露告警

当系统准备向渠道发送消息时：

- 检测手机号、身份证号、邮箱、API Key、Token、内部地址等敏感信息
- 命中后在 UI 中高亮
- 记录风险原因
- 可同步发送告警通知

### Use Case 2：出口高风险回复软拦截

当系统准备向外部渠道发送高风险内容时：

- 不立即发出
- 进入待审核状态
- 由人工决定放行、拒绝或脱敏后放行

### Use Case 3：入口黑名单拦截

当渠道消息进入系统前：

- 根据用户、群、渠道、账号、会话等字段判断是否命中黑名单
- 命中后拒绝进入业务链路
- 记录被拦截原因

### Use Case 4：入口关键词与攻击提示识别

当入口消息命中高风险关键词或攻击型提示词时：

- 标记风险级别
- 根据规则选择告警、审核或阻断

## Functional Requirements

### FR-1 风险规则模型

系统应支持以下规则类型：

- 关键词规则
- 正则规则
- 黑名单规则
- 隐私识别规则
- 结构化字段匹配规则

### FR-2 风险命中动作

系统应支持以下动作：

- 仅告警
- 标记高风险
- 阻断
- 转人工审核
- 遮罩后放行

### FR-3 命中记录

每次命中必须可追踪：

- 命中的规则
- 命中的方向（入口 / 出口）
- 命中的对象
- 命中时间
- 执行动作
- 操作结果

### FR-4 UI 可见性

系统应在当前驾驶舱中提供风险可视化信息：

- 风险事件高亮
- 风险原因说明
- 风险级别
- 审核状态

### FR-5 规则运营能力

规则应支持：

- 开关
- 优先级
- 白名单例外
- 分类
- 命中统计

## Architecture Direction

推荐采用独立的 **Policy Engine / Policy Proxy** 方案，而不是把拦截逻辑写进 OpenClaw 主工程。

### Recommended Components

1. **Routing Graph UI**
   - 展示风险、链路、审核状态、命中详情
2. **Policy Engine**
   - 执行规则匹配、风险评级、动作判定
3. **Policy Proxy / Sidecar**
   - 位于入口前或出口前
   - 负责阻断、转发、等待审核等动作
4. **Rule Store**
   - 存储规则、白名单和黑名单
5. **Audit Store**
   - 存储命中结果和操作记录

## Rollout Plan

### Stage A：只告警，不拦截

先做：

- 出口隐私泄露告警
- 风险事件高亮
- 命中记录

### Stage B：出口软拦截

再做：

- 待审核队列
- 人工放行 / 拒绝
- 审核记录

### Stage C：入口硬拦截

最后做：

- 黑名单阻断
- 攻击提示拦截
- 会话级风险限制

## Acceptance Criteria

当项目进入正式开发后，至少需要满足：

- 能识别入口和出口两个方向的风险
- 能对风险命中执行统一动作
- 能在 UI 中清楚展示命中原因
- 能记录完整审计链路
- 能独立部署，不依赖修改 OpenClaw 主工程源码

## Open Questions

以下问题需要在正式开发前确认：

1. 审核动作由谁执行？
2. 风险告警发送到哪里？
3. 规则存储使用本地文件、数据库还是远端服务？
4. 入口控制是放在渠道前，还是放在 OpenClaw 前？
5. 审核期间消息是否允许超时自动处理？

## Development Gate

本规格当前只作为公开规划文档存在。

进入正式开发的触发条件为：

- **GitHub Star > 100**

在此之前：

- 完成需求讨论
- 完成规则模型设计
- 完成 UI 和架构草案
- 不急于进入实现阶段
